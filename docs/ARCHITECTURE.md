# Архитектура проекта SQL AI Agent

## 1. Состав системы

| Компонент | Назначение |
|-----------|------------|
| **Точки входа** | CLI (`cli.py`), веб (FastAPI в `app/main.py`) |
| **Ядро агента** | `agent/` — промпты, вызов LLM, извлечение SQL, выполнение в БД |
| **Конфигурация** | `config.py` — настройки из `.env` |
| **Данные** | PostgreSQL: схема в `migrations/`, загрузка из HH.ru в `scripts/load_hh_vacancies.py` |
| **Навыки** | `skills/*.md` — текст в системный промпт (best practices) |
| **Инфраструктура** | Docker: образ приложения + контейнер Postgres |

---

## 2. Поток запроса (кто что вызывает)

```
Пользователь
    │
    ├── Веб: браузер → GET / → HTML
    │              → POST /api/ask (question) → app.main.api_ask()
    │
    └── Консоль: python cli.py "вопрос" → cli.main()
                        │
                        ▼
              agent.sql_agent.ask_agent(question)
                        │
        ┌───────────────┼───────────────┐
        ▼               ▼               ▼
   config.get_settings()   agent.prompts.build_system_prompt()
        │                        │
        │                        ├── agent.schema_desc.SCHEMA_DESC
        │                        └── skills/*.md (load_skills_text)
        │
        ▼
   OpenAI client.chat.completions.create(system, user=question)
        │
        ▼
   extract_sql_from_response(content) → sql
        │
        ▼
   run_sql(database_url, sql) → is_read_only() → psycopg execute
        │
        ▼
   return { "sql", "rows", "error" }
        │
        ▼
   Веб: JSON в ответ; CLI: таблица в консоль
```

### 2.1. Описание этапов потока

Ниже — что означает каждый этап схемы, кто его вызывает, что читает/пишет и в каком контексте это выполняется.

---

**Запуск цепочки и планировщик**

- **Веб:** Запрос приходит в контейнер приложения (uvicorn). Uvicorn работает как ASGI-сервер: один процесс, внутри — **asyncio event loop**. Каждый `POST /api/ask` обрабатывается в этом же процессе; обработчик `api_ask()` — синхронная функция, выполняется в **thread pool** (по умолчанию uvicorn запускает sync-роуты в воркерах пула), т.е. цепочка идёт в потоке из пула, не в главном потоке event loop.
- **CLI:** Пользователь запускает `python cli.py "вопрос"`. Интерпретатор Python поднимает процесс; точка входа — **typer** (декоратор `@app.command()`), который вызывает `main()`. Всё выполняется в **одном процессе, главном потоке**, без отдельного планировщика — обычный синхронный вызов по стеку.

И веб, и CLI в итоге вызывают одну и ту же функцию **`ask_agent(question)`** в `agent/sql_agent.py` — дальше цепочка общая.

---

**Пользователь → точка входа**

- **`python cli.py "вопрос"`** — пользователь запускает процесс из хоста или из контейнера (`docker compose exec app python cli.py "..."`). Аргументы парсит typer.
- **`cli.main()`** — вызывается typer’ом при разборе аргументов. Читает только переданный аргумент `question`; вызывает **`ask_agent(question.strip())`** и по результату печатает в stdout (rich Table) или ошибку и выходит с кодом 0/1.
- **Веб:** Браузер отправляет `POST /api/ask` с полем формы `question`. FastAPI передаёт тело в **`api_ask(question: str = Form(...))`**; эта функция вызывает **`ask_agent(question.strip())`** и возвращает полученный dict как JSON. Доступ к БД, файлам или секретам на этом этапе не выполняется — только приём строки и вызов агента.

---

**`agent.sql_agent.ask_agent(question)`**

Единственная точка оркестрации: вызывается из `api_ask()` (веб) или из `cli.main()` (CLI). Внутри по очереди:

1. Берёт настройки.
2. Собирает системный промпт.
3. Вызывает LLM.
4. Извлекает SQL из ответа.
5. Выполняет SQL в БД и возвращает результат.

Все следующие шаги вызываются **изнутри этой же функции** в том же потоке (веб — поток из пула uvicorn, CLI — главный поток).

---

**`config.get_settings()`**

- **Кто вызывает:** первым делом внутри **`ask_agent()`** в `sql_agent.py`.
- **Что делает:** создаёт/возвращает экземпляр **`Settings`** (pydantic-settings). При первом обращении читает переменные окружения (и при наличии — файл `.env` из текущей рабочей директории процесса). В Docker контейнере `.env` в корне проекта не монтируется автоматически; переменные задаются через `environment` в `docker-compose.yml` (или через `env_file`), т.е. процесс видит уже подставленные значения.
- **Доступ:** только чтение окружения (и опционально файла `.env`). Никаких сетевых или БД-обращений. Результат используется далее: `database_url`, `openai_api_key`, `openai_base_url`, `openai_model`, `agent_read_only`, `skills_dir`.

---

**`agent.prompts.build_system_prompt()`**

- **Кто вызывает:** **`ask_agent()`** после `get_settings()`, передаёт `settings.skills_dir`.
- **Что делает:** подставляет в шаблон системного промпта описание схемы БД и при необходимости дополняет его текстом из skills. Внутри вызывает **`agent.schema_desc.SCHEMA_DESC`** (константа, импорт при загрузке модуля) и **`load_skills_text(skills_dir)`**.
- **`SCHEMA_DESC`** — строковая константа в `agent/schema_desc.py`, описывает таблицы `areas`, `employers`, `vacancies` и связи. Никакого доступа к БД или файлам; только текст для промпта.
- **`skills/*.md` (load_skills_text):** функция в `prompts.py` принимает путь к каталогу (из настроек — обычно `.../skills`). Читает с диска все файлы `*.md` в этом каталоге, сортирует по имени, склеивает в один текст. Доступ — **только чтение файловой системы** внутри контейнера (каталог `skills/` в образе или смонтированный volume). Итоговая строка добавляется к системному промпту как блок «Дополнительные рекомендации».

Итог: один большой строковый системный промпт для LLM (схема + правила + skills). Никаких внешних сервисов или БД на этом этапе.

---

**`OpenAI client.chat.completions.create(system, user=question)`**

- **Кто вызывает:** **`ask_agent()`** после сборки промпта. Создаёт клиент `OpenAI(api_key=..., base_url=...)` из настроек и один раз вызывает `client.chat.completions.create(...)`.
- **Что делает:** HTTP **POST** на внешний URL вида `{OPENAI_BASE_URL}/chat/completions` (например, OpenAI или OpenRouter). В теле — JSON с `model`, `messages` (роли `system` и `user` с текстом вопроса). Аутентификация — по заголовку (Bearer или API key в заголовке, в зависимости от провайдера). Вызов **синхронный** (httpx под капотом у openai SDK) — поток блокируется до ответа от API.
- **Доступ:** только **исходящая сеть** из контейнера приложения к `OPENAI_BASE_URL`. Никакого доступа к БД или файлам на этом шаге. Ответ — объект с полем `choices[0].message.content` (текст ответа модели).

---

**`extract_sql_from_response(content) → sql`**

- **Кто вызывает:** **`ask_agent()`** сразу после получения ответа LLM; передаёт `content` (строка из `choices[0].message.content`).
- **Что делает:** ищет в тексте первый блок markdown вида `` ```sql ... ``` `` (регулярное выражение); если нашёл — возвращает содержимое блока; если нет — проверяет, начинается ли строка с `SELECT` или `WITH`, и в этом случае возвращает её целиком. Иначе возвращает `None`.
- **Доступ:** только к переданной строке в памяти; ни файлов, ни сети, ни БД. Результат — одна строка SQL или `None` (тогда агент вернёт ошибку «Не удалось извлечь SQL»).

---

**`run_sql(database_url, sql) → is_read_only() → psycopg execute`**

- **Кто вызывает:** **`ask_agent()`** после успешного извлечения SQL; передаёт `settings.database_url`, извлечённый `sql` и флаг `settings.agent_read_only`.
- **Что делает:**  
  - Сначала при `agent_read_only=True` вызывает **`is_read_only(sql)`**: поднимает строку в верхний регистр и проверяет, что она начинается с `SELECT` или `WITH`, и что в ней нет ключевых слов (`INSERT`, `UPDATE`, `DELETE`, `DROP`, и т.д.). Если проверка не прошла — выбрасывает `ValueError`, агент возвращает ошибку.  
  - Затем **psycopg**: `connect(database_url, row_factory=dict_row)`, в контексте соединения создаётся курсор, выполняется `cur.execute(sql)`, результат читается `fetchall()` и преобразуется в список словарей.
- **Доступ:** **исходящее TCP-подключение из контейнера приложения к контейнеру PostgreSQL** (в Docker по имени сервиса `postgres`, порт 5432). Учётные данные и хост берутся из `database_url` (из настроек/окружения). Чтение/запись только в рамках этого одного запроса; при read-only реально выполняются только SELECT’ы.

---

**`return { "sql", "rows", "error" }`**

- **Кто формирует:** **`ask_agent()`** в разных ветках: при успехе — `{"sql": sql, "rows": rows, "error": None}`, при любой ошибке (нет ключа, ошибка LLM, ошибка извлечения SQL, ошибка БД или проверки read-only) — `{"sql": ..., "rows": [], "error": "<строка>"}`. Иногда в ответ об ошибке добавляется поле `raw` (сырой ответ модели).
- **Кто использует:** вызывающая сторона — **`api_ask()`** (веб) или **`cli.main()`** (CLI). Дальше ответ не меняется, только формат вывода.

---

**Веб: JSON в ответ; CLI: таблица в консоль**

- **Веб:** **`api_ask()`** возвращает этот dict; FastAPI сериализует его в JSON и отправляет в теле HTTP-ответа с `Content-Type: application/json`. Браузер получает JSON; страница index.html (JavaScript) при необходимости рендерит таблицу по полю `rows`.
- **CLI:** **`cli.main()`** по полю `error` решает, вывести ошибку и выйти с кодом 1 или вывести через **rich** заголовок «SQL», затем таблицу по `rows` и подпись «Строк: N». Всё пишется в **stdout** процесса. Управление завершением — `sys.exit(0)` или `sys.exit(1)`.

Итог: одна и та же структура `{ "sql", "rows", "error" }` в обоих сценариях; различается только способ доставки пользователю (HTTP JSON vs stdout) и то, что в веб-сценарии один и тот же процесс uvicorn обрабатывает много запросов по очереди (или в пуле потоков), а в CLI один запуск — один запрос и выход.

## 3. Модули и зависимости

```
config.py
  └── читает .env, отдаёт Settings (database_url, openai_*, skills_dir)

agent/schema_desc.py
  └── константа SCHEMA_DESC (описание таблиц для промпта)

agent/prompts.py
  ├── импортирует SCHEMA_DESC
  ├── load_skills_text(skills_dir) — читает *.md из skills/
  └── build_system_prompt(skills_dir) — шаблон + схема + skills

agent/sql_agent.py
  ├── импортирует build_system_prompt, get_settings
  ├── extract_sql_from_response() — парсит ```sql ... ``` или SELECT/WITH
  ├── is_read_only() — проверка, что нет INSERT/UPDATE/DELETE/DDL
  ├── run_sql() — psycopg: execute, возврат list[dict]
  └── ask_agent(question) — оркестрация: settings → LLM → sql → run_sql → result

app/main.py
  ├── импортирует ask_agent
  ├── GET / → index.html
  └── POST /api/ask (Form: question) → ask_agent(question) → JSON

cli.py
  └── импортирует ask_agent; typer + rich для вывода
```

**Внешние зависимости по ходу запроса:** `.env` → PostgreSQL (миграции при старте контейнера), LLM API (OpenAI-совместимый endpoint).

---

## 4. Взаимодействие с внешними системами

| Внешняя система | Кто обращается | Когда |
|-----------------|----------------|-------|
| **PostgreSQL** | `agent/sql_agent.run_sql()` | Выполнение сгенерированного SQL |
| **LLM API** (OpenAI/OpenRouter) | `agent/sql_agent.ask_agent()` → `OpenAI().chat.completions.create()` | Генерация SQL по вопросу |
| **Файловая система** | `config` (skills_dir), `prompts.load_skills_text()` | Загрузка skills; `app/main` — чтение index.html |
| **HH.ru API** | только `scripts/load_hh_vacancies.py` | Наполнение БД (не входит в цикл запроса пользователя) |

## 5. Слои (упрощённо)

| Слой | Содержимое |
|------|------------|
| **Ввод** | CLI, веб-форма, POST /api/ask |

| **Данные** | PostgreSQL (вакансии, области, работодатели); схема из migrations |
